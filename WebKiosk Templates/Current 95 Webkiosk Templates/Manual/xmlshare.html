XML Data Sharing

This document describes special URLs that can be built to query and return results from EmbARK Web Kiosk systems. Please refer to sample template files located in the WebFolder directory of the default EmbARK Web Kiosk installation. The possible uses for this feature are limitless. A custom template may be created to generate XML output of collection data with another software tool. For example, data may be formatted in the CDWA Lite or LIDO standard formats. Such data may be exported directly to a file; or, a data harvester could access the URL and essentially import the data directly. Note: Links to XML data could be restricted to logged in authorized users.

Creating a Template |  Request |  Response 

Creating A Template

The recommended method for creating a template is to copy a sample one and rename the file. The template may be modified to meet the specific data requirements for the project. Standard conventions for the creation of XML namespaces apply but are not documented in this manual.

The templates make use of 4D Tag Syntax for the display of data. Fields from the primary table may be displayed. Additionally, data from most related tables may be displayed through the use of 4D Loops. Data may be displayed conditionally through the use of the 4DIF tag.

The type of record to display in the page. Defaults to Object records (“Objects_1”). If the “Viewed table and field names” Preference is turned on, recordType must be the user-defined file name. Otherwise, it should be the internal table name from EmbARK.

Query Parameter Format

The general format for a query strings is [field] [relation] [term]. For example, title=dog will return all records where the title is “dog”.

NOTE: We should/will support the use of quotation marks. But we don’t yet… As a result, using reserved conjunction or relation words in your term will result in an invalid search!

Field

If the “Viewed table and field names” Preference is turned on, then the field name should be the user-defined field name from EmbARK. (See “Encoded characters below.) Otherwise, the field name should be the internal field name from EmbARK.

By default, the field is assumed to be in the table from which results will be returned (see recordType parameter, above).

To query based on a field in a different table, prefix the field name with the table name, in the format [table]field. If the “Viewed table and field names” Preference is turned on, use the user-defined file name (e.g. [object]title); otherwise, use the internal table name from EmbARK (e.g. [objects_1]disp_title). After conducting the search in the related table, the EmbARK Web Kiosk will retrieve all of the linked records from the results table. For example, if recordType=object, and the query string is [artist]nationality=French, the EmbARK Web Kiosk will first query for all artists whose nationality is “French” (case-insensitive), then will return all of the object records linked to all these artists. That is, the search returns all objects that are linked to artists whose nationality is “French”.

For ease-of-use, the following special field names are also available:

Field name

Description

_ID

mfs

“MFS” is a multi-field google-like search. It will search across multiple fields as defined by the EmbARK Administrator. This option is only available for object records.

keyword

Searches through the hierarchical keyword terms applied to an object. Because keywords are hierarchical, a search for “keyword=animal” will include objects with keywords such as “bird”, “horse”, etc. This option is only available for object records.

String parameters

The following relations are available for string (alpha or text) fields:

Relation

Description

=

Exact match. Note: this relation must be percent-encoded as %3D. (See “Encoded characters” below.) For example, to do a search for title=hen, the query must be encoded in the URL as “query=title%3Dhen”.

exact

Exact match. This relation must have a space on either side. e.g. “title exact hen” is the same as “title=hen” and “title = hen”, but “titleexacthen” is invalid.

any

The term will be broken into words, then results will include records that match at least one of those words. This relation must have a space on either side. e.g. “title any hen rooster” will search for records that have “hen” or “rooster” (or both) as words in the title.

all

The term will be broken into words, then results will include only those records that match all of the term words, in any order. This relation must have a space on either side. e.g. “title all hen rooster” will search for records that have both “hen” and “rooster” as words in the title.

To do a ‘contains’ search, just append @ to your search term, percent-encoded as %40 (see “Encoded characters” below). For example, to do a search that returns both “hen” and “hens”, use “hen%40” as your search term. To return “hen”, “hens” and “then”, use “%40hen%40”.

EmbARK Web Kiosk searches are always case-insensitive.

Joining multiple search parameters

To do a query that includes multiple parameters, use the format:

[field] [relation] [term] [conjunction] [field] [relation] [term]

http://webkiosk.gallerysystems.com/results.html?layout=vg&format=xml&recordType=objects_1&query=_id=482%20or%20_id=2196

(Use the template called "vg.xml" to display Objects where the _ID equals 482 or _ID equals 2196)

There is no limit to how many parameters you may include in a query, though more parameters of course slow down the execution of the query.

Allowed conjunctions are:

Conjunction

Description

and

Both query parameters must be met.

or

Either the first or the second parameter must be met.

not

The first parameter must be met, but the second parameter must not be met.

Conjunctions must have a space on either side in order to get parsed correctly.

Note: The EmbARK Web Kiosk uses 4D Web Server technology. 4D does not treat accented characters any differently from characters without accents. So a search for “naivete” will include results of “naïveté”.

Examples

·         results.html?layout=thumbnails&query=title%3d%40girl%40

Finds all records where the title contains “girl”, and returns these results in the thumbnails.shtml template.

 

·         results.html?layout=pachyderm&format=xml&query=mfs any basket cup    or

·         results.html?layout=pachyderm&format=xml&query=mfs%20any%20basket%20cup

Performs a “quick search” with the terms “basket” and “cup”, returning all results that have at least one of these terms. A “quick search” uses pre-configured indexes to look across multiple fields which have been selected by the EmbARK Administrator for the database. The results will be formatted with the pachyderm.xml template.

 

·         results.html?layout=list&query=[artist]last name=matisse        or

·         results.html?layout=list&query=[artist]last%20name=matisse

Finds all records linked to artists where the last name is matisse. Results are returned in the list.shtml template.

 

·         results.html?layout=pachyderm&format=xml&query=keyword%3d%40birds

Finds all object records that include the keyword “birds”. This will also include objects that have keywords such as “rooster”, “ostrich” or “pigeon” which are nested under the “bird” keyword. Results will be formatted with the pachyderm.xml template.

 

Response

Available Template Variables

The following variables can be incorporated into a template using the standard syntax <!--#          4DVAR variableName-->.

 

Variable name

Description

startRecord

The first record to display. Defaults to 1.

numberOfRecords

The number of records in the result set. Defaults to 10.

nextRecordPosition

The number of the record immediately following those displayed on the current page. If there are no remaining records, returns -1. For example, if there are 25 records in the result set and records 11-20 are currently displaying, nextRecordPosition = 21.

lastRecord

The last record in the returned result set. For example, if there are 25 records in the result set and records 11-20 are currently displaying, lastRecord = 20.

atQueryFields

An array of the fields that were searched.

atQueryRelations

An array of the relations that were used to conduct the search, in the same order as atQueryFields.

atQueryTerms

An array of the terms that were searched for, in the same order as atQueryFields and atQueryRelations.

atConjunctions

An array of the conjunctions that join a given query parameter to the next query parameter (for multi-parameter searches).

pResultsTable

A pointer to the table that is being queried.

Available Template Scripts

The format for using the following scripts is <!--#4DSCRIPT/MethodName/Parameter-->.

FormatString

This script is just an entry point into 4D’s standard STRING command. See the 4D documentation for details.

 

The parameter should be a field name, formatted as [Table]Field, followed by a slash and the format string.

 

Example: <!--#4DSCRIPT/[Table]NumberField/0000--> will format the number that’s in [Table]NumberField to display as 4 digits, with leading zeroes if necessary.

 

Notes:

  • 4D’s options for formatting dates are limited to 8 options. To use this feature, use the number for the format, not the text equivalent. For example, you would use “3” instead of “Long”.

Year

This script just runs 4D’s standard YEAR OF command.

The parameter should be a date field name, formatted as [Table]DateField.

<!--#4DSCRIPT/Year/[Objects_1]_Disp_End_Date-->
